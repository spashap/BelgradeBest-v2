Fix Vercel deploy: serverless function 292MB -> 115MB (250MB limit)

The deploy failed with: function "_render" 292MB uncompressed > 250MB limit.

Diagnosis (from the locally produced .vercel/output): the admin store reads
repo files with DYNAMIC paths (join(process.cwd(), rel) in
lib/admin/store.ts getFile/listDir — the pattern the new platform module
leans on), which makes Vercel's file tracer (@vercel/nft) fall back to
including the ENTIRE working directory in the function: .git alone was
~150MB, plus public/ heroes (20MB), a traced copy of dist/client (24MB),
KB/, scripts/, package-lock — and .env, which must never ship in a bundle.

Fix: new slimServerlessFunction() integration in astro.config.mjs (same
pattern as indexNow) — after the adapter bundles the function, it prunes the
traced junk from .vercel/output/functions/_render.func. The adapter's own
excludeFiles option can't do this (exact-file matches only, no globs, as of
@astrojs/vercel 8.2.11). Kept: node_modules (GA4/GSC libs), src/ (bundled
JSON fallbacks), configs, and CRITICALLY dist/server (the function handler
is dist/server/entry.mjs — only dist/client is pruned; guarded by a comment).

None of the pruned content is needed at runtime: on Vercel all admin
reads/writes go through the GitHub Contents API (local FS is dev-only) and
static assets are served from the static output.

Verified locally: function 313MB -> 115MB, handler file present, and the
server entry module imports cleanly (smoke test). Bonus: .env no longer
ships inside the function bundle.
